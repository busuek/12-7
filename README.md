### Задание 1

**Опишите основные преимущества использования масштабирования методами:**
- активный master-сервер и пассивный репликационный slave-сервер;
- master-сервер и несколько slave-серверов.

**Ответ:**

**1. Активный master и пассивный репликационный slave (один slave)**

*Отказоустойчивость и быстрое восстановление (High Availability)*  
Пассивный slave содержит актуальную копию данных. При выходе из строя мастера slave может быть оперативно повышен до роли нового мастера. Время простоя сокращается до нескольких секунд или минут.

*Резервное копирование без нагрузки на мастер*  
Резервные копии (дампы, снепшоты) можно снимать со slave-сервера, не влияя на производительность основного мастера и не блокируя запись.

*Разгрузка мастера от аналитических запросов*  
На slave можно направлять ресурсоёмкие отчёты, выборки для бизнес-аналитики, не замедляя OLTP-нагрузку на мастере.

**2. Master и несколько slave-серверов**

*Горизонтальное масштабирование чтения (Read Scaling)*  
Несколько slave-серверов позволяют распределять входящие запросы на чтение между репликами с помощью балансировщика или на уровне приложения. Это даёт практически линейный рост пропускной способности по чтению.

*Географическое распределение*  
Slave-серверы можно разместить в разных дата-центрах или регионах, приближая данные к пользователям и снижая задержки.

*Отказоустойчивость с избыточностью*  
Наличие нескольких slave-серверов повышает надёжность: при падении одного slave’а нагрузку подхватывают остальные. В случае отказа мастера кандидатом на повышение может стать наиболее синхронизированный slave.

*Разделение нагрузок по типу*  
Разные slave-серверы могут обслуживать разные сервисы: один – для веб-приложения, другой – для внутренней отчётности, третий – для поискового движка. Это устраняет взаимное влияние разных классов нагрузки.

Таким образом, схема с одним slave ориентирована прежде всего на отказоустойчивость и резервное копирование, а схема с несколькими slave добавляет возможность масштабирования чтения, географического распределения и более гибкого разделения рабочих нагрузок.

### Задание 2

**Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц:**
- пользователи,
- книги,
- магазины (столбцы произвольно).

**Опишите принципы построения системы и их разграничение или разбивку между базами данных. Пришлите блок-схему, где и что будет располагаться. Опишите, в каких режимах будут работать сервера.**

**Ответ:**

#### Принципы шардирования

1. **Вертикальное шардирование** – разделение по функциональному признаку (разные таблицы на разных серверах).  
   - **Сервер 1 (Users DB):** только таблица `users` (профили, учётные данные).  
   - **Сервер 2 (Books DB):** только таблица `books` (каталог книг, остатки).  
   - **Сервер 3 (Stores DB):** только таблица `stores` (информация о магазинах, адреса, режим работы).  

   Это изолирует нагрузки разных доменов и позволяет независимо масштабировать каждую БД.

2. **Горизонтальное шардирование** – разбиение строк одной таблицы на несколько физических узлов (шардов) по ключу.  
   - Таблица `users` делится на N шардов по хешу или диапазону `user_id`. Шардирующий ключ: `user_id`.  
   - Таблица `books` может быть также разделена горизонтально, если предполагается очень большое количество книг и высокая нагрузка. Ключ: `book_id` или категория/издательство.  
   - Таблица `stores` обычно невелика, её можно не шардировать, а развернуть как единую референсную БД с полной репликацией.

   Таким образом, итоговая архитектура сочетает оба подхода: сначала вертикальное разделение, затем горизонтальное для наиболее нагруженных таблиц.

#### Блок-схема расположения данных

                     +------------------------+
                     |     Application        |
                     +------------+-----------+
                                  |
                     +------------v-----------+
                     |  Query Router /        |
                     |  Database Proxy        |
                     +---+--------+--------+---+
                         |        |        |
          +--------------v--+ +---v--------+ +v--------------+
          |  Users Cluster  | |Books Cluster| |Stores Cluster |
          +-------+---------+ +------+-----+ +-------+-------+
                  |                  |               |
   +--------------v-----------+     |               |
   |  Users Shard 1           |     |               |
   |  (Master + Slave)        |     |               |
   |  user_id 0..M            |     |               |
   +--------------------------+     |               |
   +--------------------------+     |               |
   |  Users Shard 2           |     |               |
   |  (Master + Slave)        |     |               |
   |  user_id M+1..N          |     |               |
   +--------------------------+     |               |
   ... (more shards as needed)      |               |
                                    |               |
                     +--------------v-----------+   |
                     |  Books Shard 1           |   |
                     |  (Master + Slave)        |   |
                     |  book_id / category      |   |
                     +--------------------------+   |
                     +--------------------------+   |
                     |  Books Shard 2           |   |
                     |  (Master + Slave)        |   |
                     +--------------------------+   |
                     ...                             |
                                       +-------------v-------------+
                                       | Stores DB (единая)        |
                                       | Master + Slave(s)         |
                                       +---------------------------+

*Пояснение:* Каждый шард внутри кластера представляет собой пару Master–Slave (или Master–несколько Slave). Приложение взаимодействует с базами через роутер/прокси (например, ProxySQL, Vitess, ShardingSphere), который по ключу шардирования направляет запрос к нужному физическому узлу. Для вертикального разделения запросы к разным таблицам адресуются сразу в соответствующий кластер.

#### Режимы работы серверов

- **Master-серверы:** принимают все операции записи (`INSERT`, `UPDATE`, `DELETE`) в рамках своего шарда или своей вертикальной БД. Обеспечивают целостность и актуальность данных.
- **Slave-серверы:** работают в режиме репликации (асинхронной или полусинхронной) от мастера. Обслуживают исключительно запросы на чтение (`SELECT`). Могут использоваться для снятия резервных копий и аналитических задач.
- **Stores DB:** если таблица `stores` не шардирована, её серверы (master + slave) обслуживают запросы к магазинам. Благодаря репликации достигается высокая доступность и возможность масштабирования чтения для этой небольшой таблицы.
- **Кластеры Users и Books:** каждый шард функционирует независимо. Отказ одного шарда затрагивает только свой сегмент пользователей или книг. Нагрузка распределяется равномерно, а количество шардов может увеличиваться по мере роста данных или трафика.
- **Роутер/Прокси:** следит за топологией кластеров, маршрутизирует запросы, может выполнять кеширование карт шардирования и автоматически исключать отказавшие узлы.

Такая комбинированная архитектура позволяет гибко наращивать производительность, сохраняя простоту разработки за счёт изоляции доменов и прозрачного для приложения шардирования.
